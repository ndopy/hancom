# [JS] getAttribute/setAttribute vs 프로퍼티(.src) — 뭐가 다를까?

## 오늘 만든 것: 클릭하면 이미지가 바뀌는 토글

`04_js/09` 실습으로 이미지를 클릭할 때마다 두 이미지 URL을 번갈아 보여주는 기능을 만들었다.

```js
const IMG_URL_A = "https://picsum.photos/96?random=1";
const IMG_URL_B = "https://picsum.photos/96?random=2";

const $picture = document.querySelector("#picture");

$picture.setAttribute("src", IMG_URL_A);

const updateImgSrc = () => {
  const currentImgSrc = $picture.getAttribute("src");
  const newImgSrc = currentImgSrc === IMG_URL_A ? IMG_URL_B : IMG_URL_A;
  $picture.setAttribute("src", newImgSrc);
};

$picture.addEventListener("click", updateImgSrc);
```

포인트:
- 이미지 URL을 상수로 분리해서 의미를 명확히 했다.
- `$picture`를 한 번만 조회해 변수에 담아두고 재사용했다 (DOM 재조회 방지).
- 클릭 핸들러를 별도 함수(`updateImgSrc`)로 분리해서 `addEventListener`에 이름으로 등록했다.

## `getAttribute`/`setAttribute` vs `.src` 프로퍼티

여기서 궁금했던 것: `img.src`처럼 프로퍼티로 직접 접근하는 방법도 있는데, 왜 `getAttribute`/`setAttribute`를 썼을까?

두 방식은 비슷해 보이지만 동작이 다르다.

- `getAttribute("src")` → HTML에 **실제로 적혀 있는 문자열 값**을 그대로 반환한다.
- `.src` 프로퍼티 → 브라우저가 **절대 URL로 변환(정규화)**한 값을 반환한다.

이번 실습에서는 두 값을 콘솔에 찍어봐도 똑같이 나왔는데, 그 이유는 애초에 `IMG_URL_A/B`가 이미 `https://...`로 시작하는 **절대 경로**였기 때문이다. 만약 이미지 경로가 `/images/a.png`처럼 **상대 경로**였다면 얘기가 달라진다.

- `getAttribute("src")` → `"/images/a.png"` (쓰여진 그대로)
- `.src` → `"https://example.com/images/a.png"` (브라우저가 도메인까지 붙여서 절대 경로로 반환)

즉, 지금 코드가 절대 경로로 문자열 비교(`===`)를 하고 있어서 우연히 두 방식의 차이가 드러나지 않은 것뿐이었다. 만약 상대 경로를 쓰면서 `.src`와 `getAttribute` 값을 섞어서 비교했다면 예상과 다른 결과가 나올 수 있다.

## DOM 프로퍼티는 get/set이 하나로 합쳐져 있다

`getAttribute`/`setAttribute`는 메서드라서 항상 괄호로 호출해야 하지만, `.src` 같은 DOM 프로퍼티는 **하나의 프로퍼티가 읽기/쓰기를 동시에 담당**한다.

```js
img.src;          // 읽기 (get)
img.src = "...";  // 쓰기 (set)
```

이 패턴은 `src`뿐 아니라 `value`, `checked`, `disabled` 등 대부분의 DOM 프로퍼티에 공통으로 적용된다. 일반 객체 프로퍼티를 다루듯이 쓰면 되고, 별도의 getter/setter 메서드를 호출할 필요가 없다는 점이 특징이다.

## 정리

- `getAttribute`/`setAttribute`는 HTML에 쓰인 원본 문자열을 그대로 다루고, `.src` 같은 프로퍼티는 브라우저가 가공(정규화)한 값을 다룬다.
- 절대 경로를 쓸 때는 두 방식의 차이가 잘 드러나지 않지만, 상대 경로를 쓰면 값이 달라질 수 있다는 걸 실험으로 확인했다.
- 실무에서는 `.src` 같은 프로퍼티 접근이 더 흔한 관례지만, "HTML에 실제로 어떤 값이 쓰여 있는가"를 그대로 다루고 싶다면 `getAttribute`/`setAttribute`도 충분히 합리적인 선택이다.
