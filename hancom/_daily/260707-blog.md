# 오늘 학습한 내용

## [JS] getAttribute/setAttribute vs 프로퍼티(.src) — 뭐가 다를까?

### 오늘 만든 것: 클릭하면 이미지가 바뀌는 토글

`04_js/09` 실습으로 이미지를 클릭할 때마다 두 이미지 URL을 번갈아 보여주는 기능을 만들었다.

```js
const IMG_URL_A = "https://picsum.photos/96?random=1";
const IMG_URL_B = "https://picsum.photos/96?random=2";

const $picture = document.querySelector("#picture");

$picture.setAttribute("src", IMG_URL_A);

const updateImgSrc = () => {
  const currentImgSrc = $picture.getAttribute("src");
  const newImgSrc = currentImgSrc === IMG_URL_A ? IMG_URL_B : IMG_URL_A;
  $picture.setAttribute("src", newImgSrc);
};

$picture.addEventListener("click", updateImgSrc);
```

포인트:

- 이미지 URL을 상수로 분리해서 의미를 명확히 했다.
- `$picture`를 한 번만 조회해 변수에 담아두고 재사용했다 (DOM 재조회 방지).
- 클릭 핸들러를 별도 함수(`updateImgSrc`)로 분리해서 `addEventListener`에 이름으로 등록했다.

### `getAttribute`/`setAttribute` vs `.src` 프로퍼티

여기서 궁금했던 것: `img.src`처럼 프로퍼티로 직접 접근하는 방법도 있는데, 왜 `getAttribute`/`setAttribute`를 썼을까?

두 방식은 비슷해 보이지만 동작이 다르다.

- `getAttribute("src")` → HTML에 **실제로 적혀 있는 문자열 값**을 그대로 반환한다.
- `.src` 프로퍼티 → 브라우저가 **절대 URL로 변환(정규화)**한 값을 반환한다.

이번 실습에서는 두 값을 콘솔에 찍어봐도 똑같이 나왔는데, 그 이유는 애초에 `IMG_URL_A/B`가 이미 `https://...`로 시작하는 **절대 경로**였기 때문이다. 만약 이미지 경로가 `/images/a.png`처럼 **상대 경로**였다면 얘기가 달라진다.

- `getAttribute("src")` → `"/images/a.png"` (쓰여진 그대로)
- `.src` → `"https://example.com/images/a.png"` (브라우저가 도메인까지 붙여서 절대 경로로 반환)

즉, 지금 코드가 절대 경로로 문자열 비교(`===`)를 하고 있어서 우연히 두 방식의 차이가 드러나지 않은 것뿐이었다. 만약 상대 경로를 쓰면서 `.src`와 `getAttribute` 값을 섞어서 비교했다면 예상과 다른 결과가 나올 수 있다.

### DOM 프로퍼티는 get/set이 하나로 합쳐져 있다

`getAttribute`/`setAttribute`는 메서드라서 항상 괄호로 호출해야 하지만, `.src` 같은 DOM 프로퍼티는 **하나의 프로퍼티가 읽기/쓰기를 동시에 담당**한다.

```js
img.src;          // 읽기 (get)
img.src = "...";  // 쓰기 (set)
```

이 패턴은 `src`뿐 아니라 `value`, `checked`, `disabled` 등 대부분의 DOM 프로퍼티에 공통으로 적용된다. 일반 객체 프로퍼티를 다루듯이 쓰면 되고, 별도의 getter/setter 메서드를 호출할 필요가 없다는 점이 특징이다.

### 정리

- `getAttribute`/`setAttribute`는 HTML에 쓰인 원본 문자열을 그대로 다루고, `.src` 같은 프로퍼티는 브라우저가 가공(정규화)한 값을 다룬다.
- 절대 경로를 쓸 때는 두 방식의 차이가 잘 드러나지 않지만, 상대 경로를 쓰면 값이 달라질 수 있다는 걸 실험으로 확인했다.
- 실무에서는 `.src` 같은 프로퍼티 접근이 더 흔한 관례지만, "HTML에 실제로 어떤 값이 쓰여 있는가"를 그대로 다루고 싶다면 `getAttribute`/`setAttribute`도 충분히 합리적인 선택이다.

## [JS] `appendChild`를 반복 호출하지 않고 `DocumentFragment`로 한 번에 붙이기

### 1. 원래 실습 코드 방식

`04_js/11` 실습은 입력한 개수만큼 `<li>`를 만들어 리스트에 렌더링하는 기능이었다. 처음 만든 코드는 반복문을 돌면서 `$li`를 만들 때마다 바로바로 `$list`에 `appendChild`를 호출했다.

```js
const renderApples = () => {
  $list.innerHTML = "";
  const count = Number($count.value);

  for (let i = 0; i < count; i += 1) {
    const $li = document.createElement("li");
    $li.textContent = `${i}번째 🍎`;

    $list.appendChild($li);
  }
};
```

동작은 잘 되지만, 반복 횟수만큼 `$list`라는 실제 DOM 트리에 매번 `appendChild`가 일어난다는 점이 마음에 걸렸다. `$li`를 만들 때마다 화면에 즉시 반영되는 구조라 반복 횟수가 늘어날수록 DOM에 손을 대는 횟수도 늘어난다.

### 2. 내가 하려고 했던 방식

원래 하고 싶었던 건, `$li`들을 반복문 안에서 다 만들어 놓고, 반복이 끝난 뒤에 딱 한 번만 `$list`에 붙이는 것이었다. 템플릿 리터럴로 `"<li>...</li>"` 같은 문자열을 이어 붙여서 마지막에 `innerHTML`에 한 번만 대입하는 방식은 익숙했지만, `createElement`로 만든 건 문자열이 아니라 객체라서 어떻게 모아뒀다가 한 번에 붙여야 할지 감이 안 잡혔다.

답은 `DocumentFragment`였다. 실제 `$list` 대신 임시 컨테이너인 `$fragment`에 `$li`들을 먼저 다 모아두고, 마지막에 `$fragment` 자체를 `$list`에 한 번만 `appendChild`하면 된다.

```js
const renderApples = () => {
  $list.innerHTML = "";
  const count = Number($count.value);
  const $fragment = document.createDocumentFragment();

  for (let i = 0; i < count; i += 1) {
    const $li = document.createElement("li");
    $li.textContent = `${i}번째 🍎`;

    $fragment.appendChild($li);
  }

  $list.appendChild($fragment);
};
```

`renderCountdown`(카운트다운 렌더링 함수)도 `while` 반복문 안에서 동일하게 `$fragment`에 `$li`를 모으고, 반복이 끝난 뒤 `$list.appendChild($fragment)`를 한 번만 호출하도록 똑같은 패턴으로 바꿨다.

### 3. `DocumentFragment`의 개념 및 장점

`DocumentFragment`는 실제 문서(DOM 트리)에는 속하지 않는 가벼운 "임시 그릇" 노드다. 특징은 다음과 같다.

- `document.createDocumentFragment()`로 만들며, 그 자체는 화면에 렌더링되지 않는다.
- `$fragment.appendChild(...)`를 아무리 여러 번 호출해도 실제 DOM 트리 밖에서 일어나는 일이라 리플로우/리페인트가 발생하지 않는다.
- `$list.appendChild($fragment)`처럼 fragment를 실제 노드에 붙이는 순간, fragment 안의 자식 노드들만 옮겨져서 `$list`에 삽입되고 fragment 자신은 비워진다. 즉 "fragment 통째로" 붙는 게 아니라 내용물만 이동한다.
- 결과적으로 반복 횟수만큼 여러 번 DOM을 건드리는 대신, 실제 DOM 조작은 마지막 `appendChild` 한 번으로 끝난다.

### 4. 결론

- 반복문 안에서 실제 DOM 노드(`$list`)에 매번 `appendChild`하면 반복 횟수만큼 DOM 조작이 발생한다.
- `DocumentFragment`를 중간 컨테이너로 사용하면, 반복 중에는 fragment에만 쌓아두고 실제 DOM에는 마지막에 한 번만 반영할 수 있다.
- 템플릿 리터럴로 문자열을 이어 붙여 `innerHTML`에 한 번에 대입하는 방식과 목적은 같지만(=DOM 조작 최소화), `createElement`로 만든 실제 요소 객체를 다뤄야 할 때는 `DocumentFragment`가 그에 대응하는 방법이다.

## [JS] `setTimeout` + `clearTimeout`으로 만든 디바운스(debounce) 패턴

### 1. 원래 실습 코드: 빈 값일 때 메시지 자동으로 지우기

`04_js/13` 실습은 과일 이름을 입력받아 목록에 추가하는 기능이었다. 입력값 없이 추가 버튼을 누르면 안내 메시지를 띄우고, 3초 뒤에 자동으로 사라지게 만들고 싶었다.

```js
const addFruit = () => {
  if (!$fruit.value) {
    $message.textContent = "과일을 입력하세요!";
    clearMessage();
    return;
  }

  fruits.push($fruit.value);
  $fruit.value = "";
};

const clearMessage = () => {
  setTimeout(() => {
    $message.textContent = "";
  }, 3000);
};
```

`setTimeout` 자체는 정상적으로 3초 뒤 메시지를 지웠지만, 한 가지 놓친 부분이 있었다. 입력값 없이 버튼을 연달아 여러 번 누르면 누를 때마다 `setTimeout`이 새로 하나씩 등록된다. 각 타이머가 결국 같은 내용(`""`)으로 지우는 것이라 눈에 보이는 오작동은 없었지만, 먼저 등록된 타이머가 먼저 발동해버려서 "마지막 클릭 기준 3초 뒤에 사라진다"는 기대와 다르게 동작할 수 있는 여지가 있었다.

### 2. 개선: 두 가지를 손봤다

#### 2-1. 타이머 ID를 저장해두고 재등록 시 이전 타이머 취소

타이머 ID를 바깥 스코프 변수에 저장해두고, `clearMessage`가 다시 호출될 때마다 기존 타이머를 `clearTimeout`으로 취소한 뒤 새로 등록하도록 바꿨다.

```js
let messageTimer = null;

const clearMessage = () => {
  if (messageTimer) {
    clearTimeout(messageTimer);
  }

  messageTimer = setTimeout(() => {
    $message.textContent = "";
  }, MESSAGE_CLEAR_DELAY_MS);
};
```

이렇게 하면 연속으로 눌러도 이전 타이머들은 실행되기 전에 전부 취소되고, 마지막 호출로 등록된 타이머 하나만 살아남아 발동한다. 즉 "마지막에 누른 시점 + 3초"에만 메시지가 사라지게 된다.

#### 2-2. 매직 넘버를 단위가 드러나는 이름의 상수로 분리

`3000`이라는 매직 넘버를 상수로 뽑아냈다. 처음에는 `TIMEOUT_SECONDS`라는 이름을 썼는데, 값 자체는 밀리초(3000)라서 이름과 단위가 맞지 않았다. 나중에 "3"으로 착각해 잘못 고칠 수 있는 이름이라 `_MS` 접미사를 붙여 단위를 명확히 했다.

```js
const MESSAGE_CLEAR_DELAY_MS = 1000 * 3;
```

### 3. 개념 정리: 디바운스(debounce)

`timerId 저장 → 재호출 시 clearTimeout → 재등록`은 "일정 시간 후 실행하되, 그 사이에 또 호출되면 대기 시간을 처음부터 다시 계산"하는 요구사항을 처리하는 표준적인 방법이며, 흔히 **디바운스(debounce)** 패턴이라고 부른다. 리액트의 `useEffect` cleanup이나 라이브러리의 debounce 유틸도 내부적으로 동일한 원리(이전 타이머 취소 + 재등록)를 쓴다.

### 4. 결론

- `setTimeout`만 단독으로 쓰면 호출될 때마다 독립적인 타이머가 계속 쌓인다. 연속 호출 상황에서 "마지막 호출 기준으로만 동작"하게 하려면 타이머 ID를 저장해뒀다가 재호출 시 `clearTimeout`으로 취소하는 처리가 필요하다.
- 시간 단위를 다루는 상수는 이름에 단위(`_MS`, `_SECONDS` 등)를 명확히 표기해야 나중에 값만 보고 착각하는 실수를 줄일 수 있다.
