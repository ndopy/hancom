# 오늘 학습한 내용

## [JS/CSS] 카드 매칭 게임 미니 프로젝트 — 트러블슈팅 노트

오늘은 `class` 문법을 제대로 써보고 싶다는 동기로 카드 매칭(짝맞추기) 게임 미니 프로젝트를 기획서 작성부터 구현, 디자인, 배포까지 진행했다. 코드는 대부분 직접 작성하고, 막힌 부분만 힌트를 받는 방식으로 진행했는데, 그 과정에서 만난 개념적인 문제와 버그들을 정리한다. (`array.lengh` 같은 단순 오탈자성 실수는 제외했다.)

### 셔플 알고리즘: Fisher–Yates

배열을 무작위로 섞는 로직을 처음 직접 구현해봤다. `array.sort(() => Math.random() - 0.5)` 같은 "트릭"도 있지만, 이 방식은 통계적으로 완전히 균등한 셔플이 아니라는 걸 알게 됐다. 대신 배열 뒤에서부터 앞으로 오면서, "아직 자리가 확정되지 않은 범위" 안에서 무작위 인덱스를 뽑아 현재 위치와 맞바꾸는 Fisher–Yates 알고리즘을 썼다.

```js
function shuffle(array) {
  for (let i = array.length - 1; i > 0; i -= 1) {
    const j = Math.floor(Math.random() * (i + 1));
    [array[i], array[j]] = [array[j], array[i]];
  }

  return array;
}
```

매번 "아직 안 섞인 구간" 중 하나를 무작위로 뽑아 확정된 자리로 보내는 방식이라, 모든 순서 조합이 동일한 확률로 나온다는 점이 sort 트릭과의 차이였다.

### Card 클래스 설계 과정에서 배운 것들

#### flip()을 토글 방식으로 바꾸면서 가드 조건도 함께 바꿔야 했다

처음에는 `flip()`을 "한 번 뒤집으면 끝"인 단방향 메서드로 만들고, 이미 뒤집힌 카드는 다시 못 뒤집게 막았다.

```js
flip() {
  if (this.#isFlipped) return; // 이미 뒤집힌 카드는 무시
  this.#isFlipped = true;
}
```

하지만 매칭 실패한 두 카드를 다시 뒷면으로 돌려야 하는 요구사항이 생기면서, "다시 덮는" 동작이 없다는 게 문제가 됐다. 그래서 `flip()`을 토글 방식으로 바꿨는데, 이때 가드 조건도 함께 바꿔야 한다는 걸 깨달았다. `isFlipped`를 기준으로 막으면 토글 자체가 안 되니, 대신 "이미 짝을 맞춘 카드는 더 이상 못 건드리게" `isMatched`를 기준으로 막아야 했다.

```js
flip() {
  if (this.#isMatched) return; // 매칭된 카드만 막는다
  this.#isFlipped = !this.#isFlipped;
}
```

상태를 하나 바꾸면 그 상태에 의존하던 다른 로직(가드 조건)도 같이 점검해야 한다는 걸 배웠다.

### DOM 업데이트와 이벤트 위임에서 마주친 것들

#### `classList.toggle`의 두 번째 인자(force)가 필요한 이유

카드가 뒤집힐 때 화면 클래스를 갱신하는 코드를 처음엔 이렇게 생각했다.

```js
target.classList.toggle("is-flipped"); // 인자 없이
```

문제는 `flip()`을 호출해도 **항상 상태가 바뀌는 게 아니라는** 점이었다. 이미 매칭된 카드에 `flip()`을 불러도 내부 상태는 그대로인데, 인자 없는 `toggle()`은 DOM의 현재 클래스 여부만 보고 무조건 반대로 뒤집어버린다. 그러면 실제 데이터(`#isFlipped`)와 화면(class)이 어긋날 수 있다. 그래서 두 번째 인자로 실제 상태 값을 넘겨서, "무조건 반대로"가 아니라 "지금 실제 상태에 맞춰 강제로 동기화"하도록 했다.

```js
target.classList.toggle("is-flipped", targetCard.isFlipped);
```

#### 이벤트 위임: `event.target`만으로는 부족해지는 순간

처음엔 카드 하나가 단순한 `<div>` 하나였어서 `event.target`이 곧 클릭된 카드였다. 그런데 3D 뒤집기 애니메이션을 위해 카드 안에 `card-inner`, `card-front`, `card-back` 같은 자식 요소를 추가하자, 이모지 텍스트를 클릭하면 `event.target`이 카드 전체가 아니라 그 안의 자식 요소가 되어버렸다. `data-order` 속성은 제일 바깥 카드 요소에만 있어서, 자식을 클릭하면 속성을 못 찾는 문제가 생겼다.

```js
const target = event.target.closest(".card");
```

`closest()`로 "클릭된 지점에서 시작해서 조건에 맞는 조상 요소를 찾을 때까지 거슬러 올라가기"로 해결했는데, 이 과정에서 새로운 케이스도 하나 생겼다. `closest()`는 조건에 맞는 조상이 없으면 `null`을 반환하기 때문에(카드 사이 여백을 클릭한 경우 등), `event.target`을 그대로 쓸 때는 없었던 `null` 체크가 추가로 필요해졌다.

덧붙여, `data-order`처럼 HTML 속성값은 항상 문자열(`"3"`)로 나오는데, `cards["3"]`처럼 배열에 문자열 인덱스를 그대로 써도 `cards[3]`과 똑같이 동작한다는 것도 이번에 알게 됐다. 배열도 결국 내부적으로는 문자열 키를 쓰는 객체이기 때문이다.

### CSS 우선순위(specificity) 충돌 — hover가 뒤집기 애니메이션을 깨버린 사례

카드에 hover 시 살짝 커지는 효과를 추가했는데, 매칭된 카드에 마우스를 올리면 회전 애니메이션이 사라지고 그냥 확대만 되는 버그가 생겼다.

```css
.card.is-flipped .card-inner {
  transform: rotateY(180deg);
}

.card:hover .card-inner {
  transform: scale(1.05);
}
```

두 선택자 모두 클래스 3개짜리라 **우선순위(specificity)가 완전히 같았다.** 이럴 땐 CSS가 파일에서 나중에 나온 규칙을 적용하는데, `transform`은 값이 합쳐지는 속성이 아니라 통째로 대체되는 속성이라 나중에 나온 `scale(1.05)`가 `rotateY(180deg)`를 완전히 덮어써버린 것이었다. `.card:hover`를 `.card:not(.is-flipped):hover`로 좁혀서, 이미 뒤집힌 카드에는 hover 효과가 아예 적용되지 않도록 해결했다.

### 몰랐던 사실: 컬러 이모지는 CSS `color` 속성을 (거의) 무시한다

카드 뒷면을 감추려고 `color: transparent`를 썼는데, 신기하게도 카드를 뒤집었을 때 `color: inherit`(검정)으로 바꿔도 이모지가 원래 색 그대로 잘 보였다. 알고 보니 컬러 이모지는 브라우저가 자체 색상 폰트(고정 색상 글리프)로 그리기 때문에 글자색(hue) 자체는 무시하지만, **투명도(alpha)**는 그대로 적용받는다. `transparent`는 alpha가 0이라 안 보이고, 검정은 alpha가 1(불투명)이라 이모지 고유의 색이 그대로 드러난 것이었다.

### 3D 카드 뒤집기: 뒷면을 미리 반대로 돌려놔야 하는 이유

카드 앞/뒷면을 `position: absolute`로 겹쳐놓고 `backface-visibility: hidden`으로 뒤집기 효과를 만들었는데, 처음엔 카드를 렌더링하자마자 이모지가 다 보이는 문제가 있었다.

```css
.card-back {
  transform: rotateY(180deg); /* 이 한 줄이 빠져 있었다 */
}
```

`.card-front`와 `.card-back`을 회전 없이(둘 다 0도) 그냥 겹쳐놓기만 하면, DOM에서 나중에 추가된 `.card-back`(이모지)이 처음부터 위에 그대로 보인다. `.card-back`을 미리 180도 돌려놔야, 카드 전체(`.card-inner`)가 나중에 180도 회전했을 때 "두 번 회전 = 제자리"가 되어 정방향으로 보이고, 반대로 `.card-front`는 그 시점에 뒤집혀서 감춰진다는 걸 이해하고 나서야 왜 이 한 줄이 꼭 필요한지 알 수 있었다.

### 자잘하지만 중요했던 것들

#### 증가 연산자 전위/후위 차이가 타이머 정확도에 영향을 줬다

경과 시간을 표시할 때 `seconds++`(후위)를 썼더니, 실제로는 1초가 지났는데 화면엔 여전히 `0`이 찍히는 식으로 항상 1초씩 밀려서 표시됐다. 후위 증가는 "지금 값을 먼저 반환하고 나서 증가"하기 때문이다. `++seconds`(전위, 먼저 증가하고 그 값을 반환)로 바꾸니 실제 경과 시간과 정확히 맞아떨어졌다.

### 배포: Netlify의 `publish` 경로는 `base` 기준 상대경로다

모노레포 구조라 저장소 전체가 아니라 `hancom/04_js/18` 폴더만 배포하고 싶어서 `netlify.toml`에 이렇게 썼다.

```toml
[build]
  base = "hancom/04_js/18"
  publish = "hancom/04_js/18"
```

배포해보니 `Deploy directory 'hancom/04_js/18/hancom/04_js/18' does not exist` 에러가 났다. `publish` 경로는 저장소 루트가 아니라 **`base` 디렉터리를 기준으로 다시 상대경로 처리**되기 때문에, 여기에 같은 경로를 또 적으면 중복돼버린다. `publish = "."`(base 자기 자신)로 고쳐서 해결했다.

### 정리

- 클래스의 상태(state) 하나를 바꾸면, 그 상태에 의존하던 다른 로직(가드 조건, 이벤트 핸들러의 동기화 방식 등)도 함께 재점검해야 한다는 걸 `flip()` 토글 전환과 `classList.toggle` force 인자 사례에서 반복해서 확인했다.
- 마크업 구조가 바뀌면(카드 하나가 단순 `<div>`에서 앞/뒷면을 가진 중첩 구조로 바뀐 것처럼) 그 구조를 전제로 짜둔 이벤트 처리 로직도 같이 손봐야 한다는 걸 이벤트 위임 사례로 배웠다.
- CSS 우선순위 동점, 컬러 이모지의 렌더링 방식, 3D 회전을 위한 사전 회전값처럼, 코드만 봐서는 예상하기 어렵고 실제 화면에서 눈으로 확인해야만 드러나는 문제들이 많았다. 브라우저에서 직접 동작을 확인하는 습관을 들여야한다.
- 배포 설정(`netlify.toml`)도 값을 그대로 옮겨 적기보다, 각 옵션이 "무엇을 기준으로 한 경로인지" 정확히 이해하고 써야 한다는 걸 배웠다.
