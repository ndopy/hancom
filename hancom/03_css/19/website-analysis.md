# 웹사이트 분석 — Apple 대한민국 (apple.com/kr)

## 선정 이유

1. 애플 같은 대기업에서 시맨틱 태그를 얼마나 잘 활용하고 있을지 궁금해서
2. 스크롤 애니메이션이나 반응형에 관련된 스킬들을 공부하는 데 도움이 될 것 같아서

분석은 실제 페이지 소스(HTML)와 연결된 CSS 파일을 직접 받아서 확인한 내용을 바탕으로 작성했다.

---

## 1. 시맨틱 태그 활용

페이지 전체에서 태그 사용 빈도를 세어보면 다음과 같다.

| 태그 | 개수 | 비고 |
|---|---|---|
| `<header>` | 0개 | **아예 사용하지 않음** |
| `<nav>` | 2개 | 글로벌 내비게이션 + 로컬 내비게이션 |
| `<main role="main">` | 1개 | |
| `<section>` | 5개 | hero, promo, gallery 등 주요 블록 |
| `<footer role="contentinfo">` | 1개 | |
| `<button>` | 14개 | |
| `role="button"` (버튼이 아닌 요소에 역할 부여) | 2개 | |
| `<ul>` / `<li>` | 26개 / 237개 | |
| `role="list"` / `role="listitem"` | 11개 / 55개 | 네이티브 리스트 대신 ARIA role로 대체한 경우 |

### 관찰한 점

- **랜드마크 태그는 확실히 사용**: `<nav>`, `<main>`, `<footer>`, `<section>` 같은 큰 구조를 나누는 태그는 명확하게 쓰고 있다.
- **`<header>`는 쓰지 않는다**: 대신 `<nav id="globalnav" role="navigation" aria-label="글로벌">` 하나로 상단 내비게이션 영역을 대체하고 있었다. 헤더 영역 전체를 감싸는 별도 태그 없이 nav 자체가 최상단 랜드마크 역할을 한다.
- **작은 단위 요소는 시맨틱 태그보다 ARIA `role` 속성으로 대체하는 경우가 많다**: 리스트(`role="list"`/`role="listitem"`)나 버튼(`role="button"`) 등을 `<ul>`/`<li>`/`<button>` 대신 `role`로 처리한 부분이 눈에 띈다. 커스텀 스타일링과 JS 인터랙션 제어의 자유도를 위해 선택한 방식으로 보인다.

**결론**: "시맨틱 태그 = 웹 접근성"이라는 공식이 곧이곧대로 적용되지는 않는다. 큰 구조(랜드마크)는 시맨틱 태그로, 세부 컴포넌트는 일반 태그 + ARIA role 조합으로 접근성을 확보하는 하이브리드 전략을 쓰고 있었다.

## 2. 반응형 & 스크롤 애니메이션

### 미디어 쿼리 브레이크포인트

```css
@media (max-width: 734px)   /* 모바일 */
@media (max-width: 1068px)  /* 태블릿 */
@media (min-width: 1069px)  /* 데스크톱 */
@media (min-width: 1069px) and (min-height: 776px)  /* 뷰포트 높이까지 고려 */
```

너비뿐 아니라 `min-height` 조건을 함께 쓰는 규칙도 있어서, 화면 비율(가로/세로)까지 고려해 레이아웃을 조정하는 것을 확인했다.

### 스크롤 애니메이션

갤러리 스크롤 연출을 담당하는 `home-gallery.built.css` 파일 하나에서:

- `@keyframes` 13개
- `transition` 50회
- `transform` 27회

이 사용되고 있었다. 애니메이션 관련 코드가 특정 파일(기능 단위)에 몰려 있는 구조였다.

CSS 변수로 애니메이션 값을 관리하는 예시도 있었다.

```css
--media-gallery-bottom-copy-duration: ...
```

지속시간(duration) 같은 값을 하드코딩하지 않고 변수로 빼서 관리하고 있었다.

> ⚠️ 이 부분은 정적 소스 분석 결과이며, 실제 스크롤할 때 어떻게 보이는지는 브라우저에서 직접 확인이 필요하다. (F12 → Elements 탭에서 스크롤하며 클래스 변화 관찰)

## 3. 색상 & 타이포그래피

### 색상 — 디자인 토큰(`--sk-` 접두사 CSS 변수) 기반 테마 전환

| 변수 | 라이트 모드 | 다크 모드 |
|---|---|---|
| `--sk-body-background-color` | `rgb(255,255,255)` 흰색 | `rgb(0,0,0)` 검정 |
| `--sk-body-text-color` | `rgb(29,29,31)` ≈ `#1D1D1F` | `rgb(245,245,247)` ≈ `#F5F5F7` |
| `--sk-body-link-color` | `rgb(0,102,204)` Apple 블루 | `rgb(41,151,255)` 더 밝은 블루 |

- 본문 텍스트가 순수 검정(`#000`)이 아니라 **`#1D1D1F`("오프블랙")** 이다. 완전한 흑백 대비보다 눈이 편한 톤을 의도적으로 선택한 것으로 보인다.
- 다크모드에서는 링크 색이 더 밝은 블루로 바뀐다. 어두운 배경 위에서 대비(contrast)를 맞추기 위한 접근성 고려로 보인다.
- `.theme-light` / `.theme-dark` 클래스에 따라 같은 변수 이름이 서로 다른 값을 갖도록 설계되어 있다. CSS 변수를 활용한 다크모드 구현의 실전 예시다.

### 타이포그래피

- `font-family`(한국어): `SF Pro KR, SF Pro Text, SF Pro Icons, Apple Gothic, HY Gulim, MalgunGothic, HY Dotum, Lexi Gulim, Helvetica Neue, Helvetica, Arial, sans-serif`
  - 자체 서체(SF Pro KR) 우선 → OS 한글 서체(맑은 고딕 등) 폴백 → 최종 sans-serif
  - `:lang(ko)` 선택자로 언어별 폰트 스택을 분기 처리 (일본어/중국어/아랍어 등도 각각 다르게 지정)
- `body { font-size: 17px }` 기준, 요소별로 `12px / 14px / 19px` 등으로 세분화. 모바일 폭(`max-width: 734px`)에서는 폰트 크기를 줄이는 규칙도 확인됨
- `font-weight`는 `400 / 500 / 600`만 사용. `700`(굵은 bold)이 없어서 전체적으로 가볍고 절제된 인상을 준다
- `:lang(ko) { word-break: keep-all }` — 한글 단어가 어중간하게 줄바꿈되는 것을 방지하는 디테일

**결론**: 색상과 폰트를 하드코딩하지 않고 디자인 토큰(CSS 변수) 체계로 관리해서 라이트/다크 테마 전환과 다국어 대응을 일관되게 처리하고 있다.

## 종합 의견

- 시맨틱 태그는 "전부 다 쓴다"가 아니라 **레이아웃의 큰 뼈대에만 적용하고, 세부는 ARIA role로 보완**하는 전략이 인상적이었다.
- 반응형은 단순 너비 기준 미디어쿼리뿐 아니라 뷰포트 높이까지 고려한다는 점이 새로웠다.
- 색상/타이포그래피를 CSS 변수 기반 토큰 시스템으로 관리하는 방식은, 지금까지 배운 CSS 변수 개념이 실무에서 다크모드·다국어 대응에 어떻게 응용되는지 보여주는 좋은 사례였다.
